Data Linking & Architecture Guide

Learn how the food database entities are structured and cross-referenced. This guide explains how to connect raw nutritional values, medical interactions, and multilingual translations in your client application.

The Central Map: metadata.json

Every consuming application must start by fetching the central map: /data/metadata.json. It lists the primary constraints that validate the rest of the database:

languages ["en", "sk"]
categories ["vegetables", "fruits", "herbs_spices", "oils_fats", ...]
tags ["green_leaves", "raw", "cooked", "affects_warfarin", "affects_coagulation", ...]

Special Indicators (Tags)

We define two special tags: affects_warfarin and affects_coagulation. When scanning the food lists in your UI, you can check for these tags to instantly render a warning symbol (e.g. ⚠️ or 🩸) next to food search results without loading the heavy interaction datasets.

Food Items & Optional Portions

Food files live under /data/categories/<category_id>.json. Each food item features standard attributes. For items where Vitamin K content is unknown (e.g., supplements), portions is set to null. For typical foods, it is an array containing amount, unit, vitamin_k_mcg, vitamin_k_100g_mcg, and optionally gram_equivalent: 

{
  "id": "blueberries_raw",
  "category": "fruits",
  "portions": [
    { "amount": 100, "unit": "g", "vitamin_k_mcg": 19.3, "vitamin_k_100g_mcg": 19.3 },
    { "amount": 1, "unit": "cup", "gram_equivalent": 148, "vitamin_k_mcg": 28.6, "vitamin_k_100g_mcg": 19.3 }
  ],
  "tags": ["berries", "k1", "raw"]
}

How Translations Are Stored

Translations live under /data/i18n/<lang>/. To display a food item (e.g., st_johns_wort) in a specific language, lookup its ID in the corresponding translation file: 

// /data/i18n/sk/herbs_spices.json
{
  "st_johns_wort": {
    "name": "Ľubovník bodkovaný",
    "alternative_names": ["Třezalka"],
    "description": "Bylinný extrakt..."
  }
}

Clinical Interaction Datasets

Detailed medical properties are decoupled from nutritional lists and stored in separate interaction files:

warfarin.json

Maps food IDs to Warfarin drug interactions. 

{
  "st_johns_wort": {
    "effect": "decreases_efficacy",
    "severity": "high",
    "mechanism": "cyp2c9_induction",
    "recommendation_key": "warfarin_avoid"
  }
}

coagulation.json

Maps food IDs to natural blood clotting influences. 

{
  "garlic_raw": {
    "effect": "anticoagulant",
    "severity": "mild",
    "active_compounds": ["allicin", "salicylates"]
  }
}

Translating Interaction Details

Values for effect, severity, and recommendation_key are clinical tags. Their human-readable localized translations are stored in /data/i18n/<lang>/common.json under the "interactions" dictionary. Always look them up there to localize your application UI.

Data Synchronization Guide

Client applications (mobile or backend) should implement a dynamic sync loop rather than downloading the entire database on every start. This reduces bandwidth and ensures instant updates.

1

Fetch the Release Manifest

Download the small `release_manifest.json` file on client start (it contains SHA-256 hashes of all database files).

2

Compare Global Version

Compare the manifest's `global_version` against the client's local version. If they match, stop (data is up to date).

3

Download Only Changed Files

If versions differ, loop through the files and download only those flagged with `"changed": true` (or check if their `sha256` hash changed compared to the locally stored file).

Why check SHA-256?

Comparing SHA-256 hashes is the most robust way to ensure file integrity. If a translation file or category file has changed, its hash will change, triggering your client code to download the fresh copy.

Example Integration (JavaScript)

async function syncDatabase(localVersion, cachedHashes) {
  const MANIFEST_URL = 'https://andrejvrabec.github.io/vitamin-k-food-database/release_manifest.json';
  
  try {
    const res = await fetch(MANIFEST_URL);
    const manifest = await res.json();
    
    // Step 1: Check if the global version differs
    if (manifest.global_version === localVersion) {
      console.log("Database is up-to-date.");
      return;
    }
    
    // Step 2: Identify changed files
    const categoryFiles = manifest.files.categories;
    for (const [categoryId, categoryObj] of Object.entries(categoryFiles)) {
      const localHash = cachedHashes[categoryId];
      
      // Step 3: Fetch file if it changed
      if (categoryObj.changed || localHash !== categoryObj.sha256) {
        console.log(`Downloading updated category: ${categoryId}...`);
        const catRes = await fetch(categoryObj.url);
        const catData = await catRes.json();
        
        saveCategoryLocally(categoryId, catData);
      }
    }
    
    // Step 4: Sync interaction files if present
    const interactionFiles = manifest.files.interactions || {};
    for (const [intId, intObj] of Object.entries(interactionFiles)) {
      const localHash = cachedHashes[intId];
      if (intObj.changed || localHash !== intObj.sha256) {
        console.log(`Downloading updated interaction: ${intId}...`);
        const intRes = await fetch(intObj.url);
        const intData = await intRes.json();
        saveInteractionLocally(intId, intData);
        saveHashLocally(intId, intObj.sha256);
      }
    }
    
    saveGlobalVersionLocally(manifest.global_version);
    console.log("Database sync completed.");
  } catch (err) {
    console.error("Failed to sync database", err);
  }
}

Versioning & Hashes

This database uses Semantic Versioning (SemVer) adaptively. Because this is a static data API, versioning is simplified to protect clients and prevent unnecessary network traffic.

Single Global Version

There are no separate version numbers inside the categories or translation JSON files. Instead, a single global version (e.g. `1.0.5`) is defined at the project level. This version matches the GitHub Release Tag (`v1.0.5`) exactly.

Automated Release Manifest

When a new release is published, the release workflow automatically generates a release_manifest.json. This file tracks the SHA-256 hash of every dataset file and flags if it changed since the previous release.

SemVer Rules for Maintainers

PATCH bump (e.g. 1.0.4 -> 1.0.5): Backward-compatible typos, spelling fixes, or value adjustments.
MINOR bump (e.g. 1.0.5 -> 1.1.0): Adding new foods, translation files, tags, or categories.
MAJOR bump (e.g. 1.1.0 -> 2.0.0): Deleting items, removing tags, renaming IDs, or modifying schemas.
